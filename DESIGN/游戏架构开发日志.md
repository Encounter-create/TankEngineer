## 一、起因：为什么需要第二次重构

### 1.1 第一次重构的成果

第一次重构（7月8日）完成了三件事：
- **SkillRenderer 统一调度**（`EffectRenderer.ts`）：14个技能自注册，main.ts 缩减 35%
- **SkillStates 统一接口**（`SkillStates.ts`）：消灭 SiegeState/PracticeState 的 50+ 字段重复
- **main.ts 收尾**：提取 `handlePracticeUI()`

成果显著，但留下了一个隐患——**重构不够彻底**。

### 1.2 触发的导火索

开发双王战争模式时，遇到了三个问题：
1. 中桥无法通过——因为 `moveTank()` 里硬编码了指挥所碰撞检测（地图中心 48px 禁区）
2. 技能效果不完整——因为 BigBang/Bivector 的 Canvas 变换只检查了 `app.siege` 和 `app.practice`
3. TwoKings 不断从 `Siege.ts` 导入函数——这违反了模式解耦原则

### 1.3 核心矛盾

**Siege.ts 既是围城模式的实现文件，又被当作"通用战斗函数库"来用**。每个新模式都需要从围城文件里 import 东西，而这些东西里又嵌着围城专属逻辑（`onEnemyKilled`、`endSiege`、CC 硬编码碰撞等）。

这不是加一个 `skipCC=true` 参数能解决的——这是架构层面的问题。

## 二、重构时间线

### 阶段 1：CC 碰撞实体化（2h）

**问题**：`Physics.ts` 的 `moveTank()` 里有 10 行硬编码代码，不管什么模式，永远在地图中心 (496, 368) 处创建一个 48px 禁入区。演习模式因此而加 `skipCC=true`，双王模式中桥也因为这个被堵死。

**解决**：把 `skipCC: boolean` 替换为 `structures?: SolidStructure[]` 参数。每个模式自己声明自己的不可通过区域。

```typescript
// 之前
function moveTank(tank, dir, dt, map, blocks, allBlocks, skipCC?)

// 之后
interface SolidStructure { pos: Vec2; radius: number; }
function moveTank(tank, dir, dt, map, blocks, allBlocks, structures?)
```

- Siege 传 `[{pos: CC中心, radius: 48}]`
- TwoKings 传 `[{蓝基地}, {红基地}, ...6塔]`
- Practice 不传（默认空）

**关键踩坑**：`sed` 删除 Siege.ts 本地函数时，从顶到底删导致行号错乱。教训：删除代码必须从底到顶。

### 阶段 2：`handleSkillActivation` 迁移（1h）

**问题**：技能激活函数在 `Siege.ts` 里，Practice 和 TwoKings 都从那里 import。

**解决**：搬入 `SkillRegistry.ts`——和 `activateSkill()` 放在一起，这是它本来该在的地方。Siege.ts 本地函数删除（-132行），改为从 SkillRegistry import。

**关键踩坑**：函数里有一处 `onEnemyKilled()` 调用（闪电技能击杀）。`onEnemyKilled` 是围城专属的击杀追踪。解决方案：去掉这个调用——被闪电击杀的敌人依然会死（`takeDamage` 生效），只是不走围城的击杀统计。

### 阶段 3：`handleBulletTankCollisions` 去 `endSiege`（0.5h）

**问题**：`handleBulletTankCollisions` 内部在玩家死亡时直接调 `endSiege(state, false)`。这个函数是围城专属的，不应该出现在通用碰撞检测里。

**解决**：删除这行调用。围城的玩家死亡检测移到 `updateSiege()` 外层——在 `updateBattle()` 返回后检查 `!state.player.alive`，触发射击特效和 `endSiege`。双王模式则在自己的 update 里把玩家死亡处理为复活。

### 阶段 4：CombatSystem 建立——搬运三大函数（3h）

**决定**：创建 `src/systems/CombatSystem.ts`，把所有通用战斗物理从 Siege.ts 搬出来。

**搬运的函数**：
- `handleBullets()`（子弹物理，含火箭/烟花/轨道弹等特殊逻辑）
- `handlePhysicsBlocks()`（物块物理，含 CC 碰撞参数化）
- `handleBulletTankCollisions()`（子弹-坦克碰撞伤害）
- `explodeRocket()`（火箭爆炸）

**CC 硬编码的参数化**：这三个函数里都有 CC 碰撞检测的硬编码。统一改为 `structures` 参数 + `onKill` 回调。

**关键踩坑**：

1. **物块击杀文字丢失**：搬 `handlePhysicsBlocks` 时漏掉了物块击杀 combo 文字设置代码（`BLOCK SMASH x2`）。后来补回。这就是"搬代码必须逐行对比"的血泪教训。

2. **Siege 和 CombatSystem 函数命名冲突**：两个文件都有 `export function handleBullets`。解决方案：Siege 用别名导入 `sysHandleBullets`，BattleEngine handler 传 sys 版本。Siege 本地函数变成死代码后删除。

3. **Practice 的 `skipCC` 参数失效**：`handleBullets` 签名从 `(state, dt, skipCC?)` 变成 `(state, dt, structures?, onKill?)`。BattleEngine 的 bullets handler 类型定义需要同步更新。忘记了就编译报错。

### 阶段 5：技能特效补全（1.5h）

**问题**：双王模式下多个技能特效缺失。

**根因分析**：
| 技能 | 缺失特效 | 根因 |
|------|---------|------|
| 陨石 | 火球/十字瞄准/文字 | Trisolaran.ts 从来没有 draw 函数，特效在 Renderer.ts 里内联画的 |
| 木马 | 友军不可见 | TwoKingsRenderer 没画 `state.allies` |
| 轰炸机 | 不可见 | TwoKingsRenderer 没画 `state.planes` |
| 二向箔 | 画面没压缩 | main.ts 的 Canvas transform 只查了 siege/practice |
| 二向箔 | 字幕重叠 | `drawBivector` 的 28px 文字和 main.ts 压缩态 22px 文字同时画 |

**修复**：
1. 创建 `drawMeteor()` 迁入 Trisolaran.ts + `registerEffect()`
2. 从 Renderer.ts 删除内联陨石渲染代码（现已由 EffectRenderer 统一处理）
3. 提取 `drawSkillEntities()` 共享函数（allies/turrets/planes/clones/fireZones）
4. main.ts 的 `bb`/`bv` 变量检查链加入 `app.twokings`
5. `drawBivector` 移除文字——全部文字的绘制权交回 main.ts（Canvas 变换上下文所在处）

### 阶段 6：`handleAllies`/`handlePlanes` 等加入 CombatSystem（1h）

**问题**：TwoKings 又从 Siege 导入了 `handleAllies` 和 `handlePlanes`。用户指出这不合理——友军 AI 和轰炸机行为是通用功能，应该抽离。

**解决**：搬入 CombatSystem。但随后用户指出更细致的分类——这些是技能实体，不该放 CombatSystem（物理层）。

### 阶段 7：SkillEntities 独立模块（0.5h）

**最终调整**：创建 `src/systems/SkillEntities.ts`，把 `handleAllies`/`handleTurrets`/`handleClones`/`handlePlanes` 四个函数从 CombatSystem 搬出。CombatSystem 回归纯物理——子弹怎么飞、物块怎么弹、碰撞怎么算。

## 四、经验教训

### 1. 搬代码必须逐行对比，不要"优化"

搬 `handlePhysicsBlocks` 时自作主张省略了物块击杀 combo 文字设置，搬 `drawQuantumCat` 时写了一个简化版小猫。两次都是用户发现效果不对才暴露。**搬移的黄金法则：git 提取原版 → 逐行粘贴 → 只改参数名 → 不做任何"顺便优化"。**

### 2. 大文件编辑优先用行号

替换 200+ 行代码块时，`Edit` 工具因 tab/空格差异匹配失败。改用 `sed -i 'start,endd'` 配合 `grep -n` 定位行号，一次成功。但 sed 删除多段代码必须从底到顶（行号从大到小），否则前序删除会改变后续行号。

### 3. 类型收紧会暴露隐藏 bug

PracticeState 之前大量使用 `string` 类型，掩盖了 `rewindReversed` 字段缺失。继承 `SkillStates` 后类型收紧为精确联合类型，`tsc` 立即发现遗漏。**接口统一不仅消除重复，还附带静态检查收益。**

### 4. `export type { X } from './Y'` 不导入

TypeScript 的这个语法只重导出类型，不使类型在当前文件内可用。内部函数仍需单独的 `import { X }`。两个语句可以共存且互不冲突。

### 5. 测试分支是最低成本的保险

整个过程在 `refactor/combat-system-extraction` 分支完成。出问题 `git checkout master` 即可恢复。这是第一次 Siege 拆分事故后建立的安全机制。

### 6. 重构不能指望一步到位

第一次重构（EffectRenderer + SkillStates）解决的是"代码在哪里"的问题。第二次重构解决的是"归属是否正确"的问题。第一次是量的压缩，第二次是质的重新分类。两次加起来才构成一个完整的架构。

## 五、未来架构建议

### 1. 渲染层的进一步共享

`Renderer.ts` 和 `TwoKingsRenderer.ts` 仍有大量重复（网格绘制、地图瓦片、坦克绘制等）。建议提取 `src/ui/BattleRenderer.ts` 作为基类：
```
BattleRenderer.ts  → drawGrid / drawTiles / drawTank / drawBullet（所有模式共用）
Renderer.ts        → extends BattleRenderer + Siege 专属（CC/HUD/debug）
TwoKingsRenderer.ts → extends BattleRenderer + 双王专属（基地/塔/木板桥）
```

### 2. AI 系统的模式参数化

`EnemyAI.ts` 的 `handleEnemyAI` 现在是 Siege 专属（目标优先级：盟友>炮塔>分身>玩家>CC）。但寻路+索敌+开火的核心逻辑是通用的。建议拆分：
```
EnemyAI.ts         → createAIContext / updateAI / shouldFire（通用）
SiegeEnemyAI.ts    → handleEnemyAI（围城专属目标优先级）
TwoKingsEnemyAI.ts → 双王专属（目标优先级：敌军>塔>基地）
```

### 3. 地图系统统一

`Map.ts` 目前只为围城服务。建议提取通用地图生成器，模式只需传入布局描述：
```typescript
interface MapLayout {
  cols: number; rows: number;
  tiles: { type: TileType; x: number; y: number }[];
  structures: { pos: Vec2; type: string; side: Side }[];
}
```

### 4. 音效系统落地

`Sound.ts` 仍是 stub。Web Audio API 的 synth 音效不需要音频文件，实现成本极低但沉浸感提升显著。
