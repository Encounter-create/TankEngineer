## 经验教训

### 1. 永远不要 git checkout 单个文件来"恢复"

这是整个事故的唯一根因。`strip-duplicates.cjs` 删除了 Siege.ts 的函数体，但同时也修改了 state 接口、export 关键字、handler 分支——这些都是脚本的副作用。`git checkout` 把这些未提交的修改全部丢弃，退回到一个比当前代码落后数十个 commit 的版本。

**正确做法**：拆之前 commit baseline，每完成一个步骤 commit。出错时用 `git revert` 或 `git reset --hard HEAD~1`，而不是 checkout 单文件。

### 2. 脚本操作必须验证作用域

`strip-duplicates.cjs` 只匹配了 `function fnName` 并删除函数体，但没有检查：
- 函数引用的模块级常量（如 `BIVECTOR_POEMS`）
- 函数依赖的接口字段（如 SiegeState 的技能字段）
- 函数的调用方是否需要重新导入

**正确做法**：脚本运行后，`tsc --noEmit` 通过只是最低标准，还需要检查每个导出的函数是否被其调用方正确引用。

### 3. 拆分的正确姿势：搬函数 + 搬常量 + 搬类型 + 搬 init

一个完整的函数拆分包括四件套：

```
原文件中需要保留的：
  ✅ 类型定义（interface SiegeState 中的字段）
  ✅ 初始化代码（createSiegeState 中的初始值）
  ✅ 模块级常量（BIVECTOR_POEMS 等）
  
目标文件中需要包含的：
  ✅ 函数体
  ✅ 函数引用的模块级常量
  ✅ 函数需要的所有 import

原文件中需要删除的：
  ✅ 函数体（保留 export 签名变为 re-export）
```

### 4. PracticeState 必须镜像 SiegeState

每次给 SiegeState 加字段，必须同步加到 PracticeState 的 interface 和 createPractice 的初始化。这是最容易遗漏的步骤，也是演习模式技能失效的第一原因。

## 下一步建议

### 方案 A：BattleEngine 抽离（推荐）

**做什么**：把 `updateSiege` 和 `updatePractice` 的共用逻辑抽成 `updateBattle()`。

**为什么现在做**：Siege.ts 刚经历过一轮拆分，结构已经清晰。通用的 bullet/physics/particle/skill update 调用链一目了然。抽离 BattleEngine 只需要把已有调用包装成一个函数，Siege.ts 和 Practice.ts 各自调用它。

**预估工时**：1 小时（纯搬移，不改逻辑）

**风险**：低。和技能拆分一样，分支操作 + 逐步骤 commit。

### 方案 B：继续做新技能

**做什么**：叶公好龙、长城烽火、空城计等。

**为什么先不做**：核心循环的问题（技能展示效率低、无叙事框架、无 PvP）比缺一个新技能更紧迫。一个叶公好龙只能让玩家新鲜 5 分钟，但 BattleEngine + 平行宇宙模式可以让 15 个技能同时发光。

### 推荐

**A 优先，B 跟进。** 抽完 BattleEngine 后，Siege.ts 预计可降到 ~800 行，Practice.ts 降到 ~150 行。然后在这个干净的架构上再做新技能，"框架 + 新内容"的组合模式正式生效。
