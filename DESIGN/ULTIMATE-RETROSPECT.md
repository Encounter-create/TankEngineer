## 一、项目全时间线

### 阶段 0：奠基（7月4日，4h）

**目标**：建立项目骨架。

**产出**：
- Canvas + TypeScript + Vite 技术栈选定
- `Vec2` 不可变向量库、`Grid` 30×22地图系统
- 6种瓦片类型（EMPTY/BRICK/METAL/WATER/GRASS/ICE/BARREL）
- `GameLoop` 固定60Hz时间步
- `Input` WASD+鼠标输入管理
- `Parts` 零件系统：炮管(2)+炮塔(2)+车身(2)+车长(0)=6个初始零件
- `TankConfig` 坦克方程约束（重量等级 light/medium/heavy）
- `Inventory` localStorage持久化
- `Physics` 碰撞检测+弹性反弹+坦克移动+冰面惯性

**关键决策**：
- 零外部依赖（纯Canvas手绘）
- 不可变Vec2（避免引用bug）
- 零件系统用"重量"约束防止最强零件堆砌

### 阶段 2：技能系统（7月5日-7日，10h）

**目标**：差异化能力，文化表达。

**产出**：
- 25个车长技能（维修/冲刺/弹幕/烟雾/空军上校/炮塔工程师/巫师/忍者/重力/时间/闪电/复苏/三体人/歌者/薛定谔/爱因斯坦/庞加莱/奇点/投影者/奥德修斯/诺亚/达摩克利斯/叶公/创世者/雷神）
- 每个技能 = update（状态机推进）+ draw（Canvas特效）+ playQuote（台词系统）
- 粒子系统（`Particle`）+ 火圈（`FireZone`）+ 伤害数字（`DamageNumber`）
- 友军系统（`AllyTank`/`CloneEntity`/`TurretEntity`/`Plane`）

**技能设计哲学**：
- "机制即视觉"——不是先做机制再加视觉，核心画面本身就是机制
- "典故优先于机制"——从文学/历史/科学故事出发设计，不是先想"做个闪电链"
- "1-2步最优"——木马（滑入→开门）、诺亚（涨潮→退潮），步骤越少越好

**核心难题**：
1. Siege.ts 膨胀到2850行——每次加技能都要在代码山找位置
2. main.ts 成为技能渲染的"收纳箱"——14个 if-else 硬编码
3. 演习模式（Practice.ts）与围城完全重复的更新/渲染循环

### 阶段 3：第一次架构重构（7月8日，3h）

**起因**：Siege.ts 2850行不可维护。添加新技能要改4个文件。

**第一次拆分**（`refactor/split-siege`）：
- 10个技能函数提取到 `src/skills/`
- Siege.ts: 2850→1538行
- **事故**：`git checkout src/modes/Siege.ts` 把脚本副作用全部丢弃，退回到落后数十commit的版本。手动恢复耗时3h。

**架构优化三步走**（`refactor/architecture-cleanup`）：

1. **SkillRenderer 统一调度**：`EffectRenderer.ts`——14个技能自注册+`renderAllEffects()`。main.ts 860→545行（-35%）。添加技能不再需要改 main.ts。

2. **SkillStates 统一接口**：`SkillStates.ts`——SiegeState 和 PracticeState 的50+字段重复合并为一处。添加新技能只改1个类型文件。

3. **main.ts 收尾**：提取 `handlePracticeUI()`。

**经验**：
- 搬代码必须逐行对比，不做"顺便优化"（物块文字+小猫两次踩坑）
- sed 删代码必须从底到顶
- 测试分支是最低成本保险

### 阶段 4：双王战争（7月8日-9日，8h）

**目标**：第二个游戏模式，验证通用框架。

**产出**：
- `TwoKings.ts` 对称3路PvE（650行）
- 河流+3桥+蓝红各1基地3防御塔
- 航线导航系统（5航点：基地→塔→桥→敌塔→敌基地）
- 分层索敌AI（敌军>玩家>本路塔>残余塔>基地）
- 目标锁定机制（`_lockId` 直到目标死亡/出视野）
- 渐进波次（第1分钟1辆/路→第2分钟2辆→第3分钟3辆）
- 3分钟游戏时长
- 基地回血（绿圈+绿粒子）
- 塔摧毁特效（爆炸粒子+震屏+中央大字）
- 水面波浪背景（诺亚式三层正弦波+浪花喷雾）

**AI开发——十次迭代**：

| v | 方案 | 失败原因 |
|---|------|---------|
| 1 | `navigateToward()` 直接设vel | 加速度惯性→弧线漂移 |
| 2 | `updateAI()` 巡逻状态机 | 随机游走不沿航线 |
| 3 | `moveDir.norm()` 直指目标 | 每帧方向变化→螃蟹走 |
| 4 | `updateAI`+`ctx.targetPos` | CHASE模式追航点 |
| 5 | 漂移消除 `vel.dot()` | 飞一样快→永远满速 |
| 6 | 网格四方向 DIR4 | 重复迭代→收敛 |
| 7 | 目标锁定 `_lockId` | 全打同一座塔 |
| 8 | `towers[lane]` | 塔灭后贴脸基地 |
| 9 | 基地加入索敌 | 边界不开火 |
| 10 | `fireR-10`停止+`fireR*0.5`后退 | ✅ |

**教训**：**自造简化版是效率黑洞**。前十次迭代的前五次全在自造AI，都失败。第六次直接用 Siege 的网格四方向+速度钳制+doFire保持距离，一次成功。

## 三、经验教训总汇

### 工程类

1. **搬代码黄金法则**：git提取原版→逐行粘贴→只改参数名→不做"顺便优化"。违反=返工。

2. **sed删代码从底到顶**：行号从大到小删，否则前面删除改变后续行号。

3. **`export type { X } from` 不导入**：只重导出类型。内部引用需单独 `import { X }`。

4. **类型收紧暴露隐藏bug**：PracticeState 用 `string` 掩盖了 `rewindReversed` 缺失。精确联合类型→tsc立即发现。

5. **测试分支最低成本保险**：重构全在测试分支做，出问题 `git checkout master` 即恢复。

### 调试类

6. **排查3次未定位必须加 console.log**：多坦漂移反复6次盲改，一行log10秒定位。

7. **性能优化先测再改**：封面卡顿4次乱试，最后定位到 `handleBullets` 太重。

### 设计类

8. **自造简化版是效率黑洞**：封面动画+双王AI都证明了这一点。直接用成熟系统组装。

9. **机制即视觉**：木马（滑入放兵）、诺亚（水涨水退）、陨石（砸下来）、二向箔（压扁）。**最好的技能一帧画面就能讲清楚。**

10. **典故是加分项，不是理解前提**：美杜莎不认识也能懂"石化"，耶利哥城墙不认识就不知道"墙为什么塌"。

### 架构类

11. **重构不能一步到位**：第一次解决"代码在哪里"，第二次解决"归属是否正确"。量的压缩+质的分类。

12. **新模式安装清单**：通用框架提供7个模块（CombatSystem/SkillEntities/SkillRegistry/EffectRenderer/BattleEngine/BattleUI/Physics），新模式只写AI/规则/Renderer。

## 四、项目全貌评估

### 设计理念

坦克工程师有三重身份：

**1. 几何物理对战（骨架）**
49个零件×4维度组合→重量方程约束→9种弹道物理风格。这不是"数值换皮"——直射管和磁轨管的差异是物理规律的差异（反弹 vs 沿金属滑行），不是"伤害+5"。

**2. 文化表达（灵魂）**
"典故优先于机制"——25个车长技能里，二向箔的压扁、木马的金光、陨石的"世界属于三体"、大洪水的波浪，都是可以直接截图的视觉名片。这是与市面上所有坦克游戏的本质差异。

**3. 演出体验（情感）**
Canvas全屏特效+粒子系统+震屏+台词。技能不是"播放动画"，而是改变整个世界的状态——二向箔让世界压扁，要有光从全黑中恢复，引力透镜扭曲画面。玩家按E的瞬间产生"卧槽"。

### 核心玩法

**优势**：
- **零件组合产生真正策略深度**：轻量级玻璃大炮 vs 重量级移动堡垒，不是数值上的二选一，而是玩法上的根本不同
- **9种弹道各有独立物理规则**：反弹衰减/穿透减半/抛物线/烟花子母弹/轨道双星/火箭追踪/散射扇形/磁轨滑行——每换一把炮管，战斗方式完全不同
- **14个技能是文化名片**：不是"闪电链""火球术"的换皮模板。每个技能都有独立的状态机+视觉语言+台词
- **双王模式验证了框架复用性**：从零到可玩约8h，AI/物理/技能全部零代码复用

**问题**：
1. **经济系统形同虚设**：所有零件初始拥有+50000金币。无任何收集驱动。玩家打开游戏第一秒就拥有全部内容。
2. **零件决策深度扁平**：49个零件在4个独立维度上选，维度间缺乏交叉影响。仅有的2组协同效应（Synergy.ts）是隐藏的，玩家不查文档不知道。
3. **游戏模式之间缺乏叙事串联**：玩家在大厅随机选模式→打完回大厅→再随机选。无任何 progression 或 motivation。
4. **无任何教程/引导**：新玩家打开不知道做什么。全靠快捷键提示条和试错。
5. **音效合成单调**：Web Audio API synth 可用但远不如真实音效文件丰富。

### 底层机制

**优势**：
- 物理引擎扎实：碰撞检测圆角矩形+弹性反弹+方块推动+惯性滑行+冰面零摩擦
- 通用对战框架干净：7个模块各司其职，接口清晰（SolidStructure/OnKillFn/structures参数）
- 技能系统完备：25个技能 update+draw+EffectRegister 三件套。新增技能只改2个文件
- 渲染管线清晰：EffectRenderer 统一调度，drawSkillEntities 共享实体绘制
- AI系统可复用：EnemyAI 的 patrol/chase/fire 状态机同时服务于围城/双王/封面/Exercise

**问题**：
1. **渲染层未完全共享**：Renderer.ts 和 TwoKingsRenderer.ts 仍有网格/瓦片/子弹的重复绘制（约150行）
2. **地图系统模式绑定**：Map.ts 只为围城服务。双王地图在 TwoKings.ts 里手写。封面地图在 Demo.ts 里手写。
3. **AI系统未参数化**：EnemyAI 的 patrol/chase/fire 状态机目标锁定逻辑是为围城设计的（CC中心→巡逻→追玩家）。双王和封面都绕过了它，各自写了自己的目标选择。
4. **网络/多人架构零准备**：无状态同步抽象，无输入序列化。

### 最大亮点

**1. 技能的文化表达**——这是 TankEngineer 真正的护城河。

市面上不缺酷炫特效。但"用典"和"玩梗"——把荷马史诗、希腊神话、量子力学、北欧神话翻译成游戏事件——这件事极少有游戏在做。玩家可能记不住第几个车长是干嘛的，但他们会记住"那个召唤木马的"、"那个压扁画面的"、"那个洪水淹没一切的"。

**2. 对战框架的通用性**——这是后续开发效率的保证。

两次重构建立的四层架构（CombatSystem物理+SkillEntities实体+SkillRegistry触发+skills演出）使得双王模式从零到可玩约8h。未来任何新模式都能以"导入通用模块+写自己的AI规则"的方式组装出来。

**3. 几何物理的玩法深度**——这不是"好看"，是"好玩"。

9种弹道的物理差异意味着每种炮管都是一套不同的瞄准思路。反射管需要计算反弹角（2次反弹×0.8衰减→最佳角度不是正对敌人），磁轨管需要利用金属墙做导轨，烟花管需要在飞行中不断调整母弹位置。

### 不足之处

1. **无经济循环**：这是最大的短板。玩家没有"追求"——一切内容一开始就有了。
2. **无叙事框架**：为什么一个游戏里同时有特洛伊木马和量子叠加态？需要世界观设定（如"时间线崩塌，所有文明的力量同时降临"）。
3. **无社交/竞技**：纯单机。没有排行榜、没有分享、没有PvP。
4. **移动端不支持**：Canvas 960×704 + 键盘操作。触摸屏完全无法游玩。
5. **AI不够智能**：所有敌人共用一套AI（patrol→chase→fire）。没有差异化行为，没有Boss设计。

### 未来展望

**短期（1-2周）——打磨MVP**：
- 音乐接入（等用户上传文件）
