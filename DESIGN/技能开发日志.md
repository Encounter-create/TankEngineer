### 确立的审美标准

四轮讨论沉淀出的 TankEngineer 技能设计铁律：

1. **一句话画面**：能不能用一句话描述这个技能的核心视觉？（"一个东西砸下来"、"水涨起来淹没一切"、"画面压扁"、"黑暗→光扩散"、"锤子追击鼠标"）

2. **2 步以内**：从释放到视觉高潮，逻辑步骤 ≤ 2？超过 3 步的设计，砍。

3. **不依赖典故**：不认识这个典故的人能不能仅从视觉理解发生了什么？典故是加分项，不是理解前提。

4. **视觉即机制**：特效不只是"好看"，而是视觉变化本身就是游戏效果。光晕扩散 = 清场，水位变化 = 淹死，不是"先结算伤害再放个特效动画"。

5. **不设叙事代理**：能不能不需要角色/文字来"解释"发生了什么？木马不需要有人解说，黑暗和光明自己会说话。（"要有光"后来去掉天使正是这一原则的体现。）

> **核心公式：机制就是视觉，视觉就是故事。最好的技能是你画一帧关键画面给没玩过游戏的人看，他看了一眼就能说出这是什么故事。**





### 设计初衷

圣经《创世纪》——"神说，要有光，就有了光。"这是人类文明中最具辨识度的句子之一。设计的核心是一个视觉母题：**黑暗降临 → 光的出现 → 世界被净化**。机制就是视觉本身——黑屏消退的过程就是清场的过程。

### 开发时间线

#### 第一阶段：天使版本（废弃）

**初始设计**：画面渐黑 → 天使（翅膀+头环，参照木马中奥德修斯风格）从顶部降下 → 宣告 "There shall be light" → 天使升回 → 火苗燃起 → 光圈扩散 → 清场恢复。

**为什么废弃**：用户测试后给出明确判断——"小人的引入使画面变得很不纯粹"。天使是一个叙事元素，但它的存在让观众的注意力从"黑暗→光明"的核心视觉母题上转移到了"这个小人是谁"的无关问题上。设计不简洁。

**教训**：技能不是动画片。如果一个视觉元素不能直接服务于核心机制，就去掉。要有光的核心是光与暗的对抗，不需要中介者。

#### 第二阶段：纯光影版本（最终版）

去掉天使后，技能简化为纯粹的黑暗→光明的过渡：

```
黑暗 → 文字 → 火苗 → 光扩散 → 世界净化
```

没有任何中间人、没有叙事代理、没有"角色"。观众直接面对从全黑中逐渐恢复光明的画面——这本身就是最原始、最强有力的视觉体验。

#### 第三阶段：精确的时间线校准

用户对节奏有非常精确的要求。经历了三轮调整：

**v1（粗略版）**：0-2s 黑场渐入，3-5s 文字，5s 清场+火苗，5-9s 光圈扩散。

**v2（去掉天使，两阶段）**：darkening(0-5s) → ignition(5-9s)。

**v3（最终精确版，用户逐秒指定）**：

| 时间 | 画面 |
|------|------|
| 0-2s | 黑场渐入 |
| 2-3s | 全黑保持 |
| 3-5s | "There shall be light" 金色文字浮现（静态，不脉动） |
| 5-6s | 文字逐渐褪去 |
| 6s | 清场（敌人+物块全消），画面全黑 |
| 6-7s | 全黑保持 |
| 7s | 火苗在中央燃起 |
| 7-8s | 光圈从小点扩张为小圆（半径120px） |
| 8-9s | 停顿，保持小光圈 |
| 9-12s | 光圈从小圆撑满全屏 |
| 12-15s | 火苗逐渐隐退，游戏恢复光明 |
| 15s | 技能结束 |

**关键细节**：文字"不脉动"是用户特别强调的。初版文字有 `sin` 脉动效果，用户指出这破坏了庄严感——这是圣经中最神圣的句子，应该沉稳、安静地出现。去掉脉动后文字的静默确实更有力量。

### 工程实现

**核心技术**：`ctx.createRadialGradient` 做光晕扩散。

```typescript
const grad = ctx.createRadialGradient(cx, cy, innerR, cx, cy, r);
grad.addColorStop(0, 'rgba(0,0,0,0)');    // 圆心透明（可见游戏画面）
grad.addColorStop(0.6, 'rgba(0,0,0,0)');  // 过渡带保持透明
grad.addColorStop(1, 'rgba(0,0,0,1)');    // 边缘全黑
ctx.fillStyle = grad;
ctx.fillRect(0, 0, MAP_W, MAP_H);
```

这个渐变天然产生"亮度由圆心向边界递减"的效果——fire 处的画面是明亮的，向外逐渐过渡到黑暗，过渡带宽度设为 `min(80, r*0.4)` 使过渡在小光圈时锐利、大光圈时柔和。

**火苗绘制**：5 层火焰舌（白→橙→红，正弦位移 + 随机尺寸），外围径向光晕，核心白色亮点。

**清场**：在 6s 时遍历 `state.enemies` 全灭 + `state.physicsBlocks` 全灭，爆炸粒子 + 音效。清场时机在黑暗中进行，玩家看不到过程——"黑暗中的净化"，这正是圣经叙事的核心：光出现时，旧世界已经不存在了。

**黑屏渲染层次**：drawGenesis 必须在 main.ts 中**最后**调用（在所有游戏渲染之后），否则黑屏会被后续渲染覆盖。

### 经验教训

#### 1. 视觉母题驱动，不要叙事代理

天使被砍掉是正确决定。"要有光"不需要一个角色来告诉观众发生了什么——黑暗和光明自己会说话。好的技能设计让视觉本身成为叙事，不需要翻译者。

#### 2. 节奏感需要精确到秒

v1 粗略、v2 两阶段、v3 六阶段——每一轮都在增加时间精度。9 秒内的每一个停顿（清场后的 1s 黑屏、小光圈的 1s 暂停）都不是空白，而是呼吸。9-12s 用 3 秒撑满全屏（而非 1 秒瞬间完成），给观众时间感受"光在扩散"的过程。

#### 3. 细节决定基调

文字不脉动——这是用户坚持的微观决策。脉动 = 活泼、游戏感；静态 = 庄严、仪式感。对"要有光"这个主题，静态是唯一正确的选择。

#### 4. 清场时机在黑暗中进行

如果清场在明亮中执行（如 v1 设计），玩家会看到敌人瞬间消失——像 bug。在 6s 全黑中清场、7s 才燃起火苗，玩家只看到"光明恢复后世界已净化"。黑暗屏蔽了清理过程，保持了神秘感和神圣感。



### 设计初衷

北欧神话中托尔的短柄锤 Mjolnir，掷出后可自动追击敌人并返回持有者手中。不同于"陨石"的一次性砸落或"木马"的固定轨迹，锤子应该是一个**有物理自主性的独立物体**——始终在加速、始终在运动、有惯性、会旋转、有巨大击退力。

最初的设计方案是鼠标驱动：锤子从左上角飞入，追击鼠标位置，鼠标静止时在鼠标处上下弹跳悬浮。但很快发现这个思路行不通。

### 开发时间线

#### 第一阶段：鼠标追踪模型（失败）

**实现**：锤子从 `(-100,-100)` 加速飞入，进入 `active` 阶段后持续向鼠标位置加速（300 px/s²），无速度上限。鼠标静止时切换为悬浮模式（重力+反弹驱动的上下浮动）。

**遇到的困难**：
1. **转向问题**：锤子绘制时锤头在顶部（-Y），但 `atan2(vel)` 返回的角度直接应用后锤头不朝向飞行方向。修复：角度偏移 `+ PI/2`，使锤头正确指向运动方向。
2. **转向瞬切**：锤子角度瞬间跳变，缺乏物理感。修复：引入角速度 `PI × 8 rad/s`，使用 `normalizeAngle` + `maxStep` 模式（与坦克炮塔一致），锤子转弯时产生自然的甩尾感。
3. **锤子失控**：鼠标刹车圈添加了 `approachSpeed > 0` 的条件判断，导致锤子飞过鼠标的瞬间刹车立即关闭，锤子直接飞走无法控制。修复：移除方向判断，只要在圈内就始终拉向鼠标。

**为什么彻底放弃鼠标模型**：即使修复了上述问题，锤子仍然无法在鼠标处稳定下来。用户给出了精准的物理判断——_这是一个保守场问题_。

#### 第二阶段：保守场的困境（物理洞察）

用户的分析（原文）：

> "从物理的眼光看，玩家是力场的中心，那么锤子天然是做椭圆运动而不是追踪玩家并在玩家身边停下来。如果把玩家向周围释放的力场看作保守场，那么如果玩家不动，锤子的机械能不变，锤子就始终无法停下来。而且随着玩家的运动，锤子的势能可能因为锤子与玩家距离增加而突然增加，这就导致锤子运动到玩家周边时的速度越来越快。"

这个分析极其精准。核心问题：

- **保守场中机械能守恒**：纯向心加速度无法耗散能量，锤子只能在椭圆轨道上永恒运动，或者越飞越快。
- **刹车圈仍是保守场**：即使在 120px 内施加 100 倍加速度，它仍然是保守力——只是力常数更大，不改变能量守恒的本质。
- **结论**：必须引入**非保守力**（耗散力）来打破能量守恒。

#### 第三阶段：轨道环 + 弹簧力 + 径向阻尼（成功）

**核心设计**——不在玩家身上设力心，而在玩家周围创建**稳定轨道环**：

```
三个力同时作用：

1. 弹簧力 F = SPRING_K × (dist - ORBIT_R)
   - dist > 55px → 力向内（拉回轨道）
   - dist < 55px → 力向外（推开，防止贴死）
   - dist = 55px → 力为零（平衡点，锤子自然悬浮）

2. 径向阻尼
   - 仅耗散径向速度分量（vel 在 toward 方向上的投影）
   - 保留切向速度 → 锤子可以在轨道环上自由绕行
   - 振荡逐步收敛到平衡点

3. 远距追击（>480px）
   - 1x 加速度自由追击，保持"越飞越快"的动态视觉
```

**为什么这个方案成功**：
- 弹簧力是保守力，但径向阻尼是非保守力——两者结合，系统自动收敛到 55px 轨道环
- 切向速度不受阻尼影响 → 锤子可以在轨道上持续绕行（hovering around）
- 玩家移动时轨道环跟随，锤子追着轨道跑
- 捕获圈 480px（后来调大）让锤子从几乎全屏范围被拉向轨道

**参数演进**：

| 迭代 | CAPTURE_R | SNAP_ACCEL | SPRING_K | 设计思路 |
|------|-----------|-----------|----------|---------|
| v1 | 80 | 6000 (20x) | — | 简单刹车圈 |
| v2 | 240 | 30000 (100x) | — | 加力刹车 |
| v3 | — | — | 46 | 弹簧轨道环 |
| v4 | 480 | — | 7 | 扩大捕获圈，比例调K |

关键教训：**参数应该按物理量纲推导**。`SPRING_K = 目标边缘力 / (CAPTURE_R - ORBIT_R)`，而不是拍脑袋定值。

#### 第四阶段：视觉特效

1. **暴风雨+闪电**：完全复用 Noah.ts 的代码——斜向雨滴粒子 + 2-5 段随机折线闪电 + 白色全屏闪 + 金色粒子。持续 5s（entering + active 前 3s），之后雨量减半，闪电停止。

2. **托尔小人**：参照奥德修斯风格绘制——带翼北欧头盔（穹顶+护鼻+眉弓+后掠小翼）、红披风、银灰铠甲、持锤右臂。锤子入轨时悬浮锤子上方，2s（后改5s）后向上升空离场。

3. **台词系统**：7 套托尔经典语录随机呈现，金色文字逐行显示在托尔头顶。第一次迭代只显示 2s 太短，延长到 5s。

4. **头盔重绘**：初版头盔是简单圆弧+宽大羽毛翼，用户评价"丑"。重绘为北欧风格——弧形穹顶、眉弓金色饰带、鼻部护条、小巧后掠翼带内羽线。

### 经验教训

#### 1. 保守场分析必须先于代码

这是整个开发中最重要的教训。涉及复杂物理交互时，**先做力学分析**：
- 识别力场类型（保守/非保守）
- 如果是保守场 → 物体永远无法停下来 → 必须引入耗散力
- 力场中心设计（单点吸引 vs 轨道环平衡点）

如果在写代码之前做这个分析，可以跳过鼠标模型的两个迭代（刹车圈、方向判断），直接进入轨道环方案。

#### 2. 阻尼要有方向性

全局阻尼 `vel *= scale` 会让锤子完全停下来，失去轨道绕行的动态美。**径向阻尼**只耗散径向分量、保留切向分量，锤子收敛到轨道后仍能自由绕行——既稳定又动态。

#### 3. 弹簧平衡点 > 强力吸引

"100 倍加速度吸住锤子"的思路本质上是和物理对抗。弹簧力 `F = k(dist - orbitR)` 的思路是**利用物理**——创建一个力学平衡点，让系统自动收敛。前者需要不断调参，后者天然稳定。

#### 4. 参数按量纲推导，不拍脑袋

`SPRING_K = 3000 / 65 ≈ 46` 是从"边缘拉力 ≈ 10x 基础加速度"推导出来的。改 CAPTURE_R 时同步调整 `K = 3000 / 425 ≈ 7`。每个参数都有物理含义，调参时可预测结果。

#### 5. 复用现有视觉语言

暴风雨闪电完全复用 Noah.ts，托尔小人参照 Trojan.ts 的奥德修斯风格，台词复用 playQuote 系统。新技能的视觉特效应尽可能从已有资产中"组装"而非"新建"。

## 2. 📐 二向箔 — 歌者 (Bivector Foil)

### 设计需求

### 困难与解决
| 困难 | 解决 |
|------|------|
| shear=3.0 时画面被推到屏幕外不可见 | 降到 0.8，上边缘偏移 ~192px 正好可见 |
| transform 放在 renderSiege 内无效 | 移到 `main.ts` 的 `render()` 全局层，在 `clearRect` 后立即应用 |
| 白屏 8s + 恢复 5s 完全不执行 | `bv` 变量只判断 `=== 'compressing'`，改了 `!== 'idle'` 但 `?.` 运算符造成 null→true 的陷阱 |

## 4. 🌀 时空曲率 — 爱因斯坦 (Spacetime Curvature Lens)

### 设计需求

|------|------|
| 同心环版本只有均匀缩放，无扭曲感 | 改用 `getImageData` 逐像素位移，每个像素的位移量与到中心距离非线性相关 |

## 5. ⏪ 庞加莱回归 — 时间倒流 (Poincaré Recurrence)

### 设计需求
- 5 秒时间倒流：子弹飞回、敌人退行、粒子倒流
- 蓝色滤镜 + 残影拖尾
- 倒流结束熵爆发：蓝色冲击波毁灭一切

### 工程落地

- **控制点 1**：`updateRewind` 在每帧**最前面**反转所有实体速度（只反转一次，`rewindReversed` 标记防振荡）
- **控制点 2**：`handleEnemyAI` 在 rewind 期间 `return` 跳过（防止 AI 覆写反转速度）
- **控制点 3**：子弹制导/烟花分叉在 rewind 期间跳过
- 粒子倒流：`p.life += dt` 生命逆增长，`p.vel *= -1`

### 困难与解决
| 困难 | 解决 |
|------|------|
| 技能释放后所有实体静止不动 | `updateRewind` 最初放在循环**末尾**，反转发生在物理/AI 之后，等于无效。移到循环最前面 |
| 实体原地振荡不倒退 | 每帧反转 `vel *= -1` 导致 `+50→-55→+60→-66` 来回摆动。加 `rewindReversed` 标记只反转一次，后续帧加速同向 |
| AI 覆写反转速度 | 反转后的 `enemy.vel` 被 `handleEnemyAI` 立即覆写成正常方向。AI 函数开头加 `if (rewindPhase !== 'idle') return` |
| 摩擦力让速度很快降为零 | 反转后 `moveTank` 的 `FRICTION` 照常衰减速度。每帧强制维持最低速度（敌人 40、子弹 100、方块 30 px/s）+ 持续加速 `scale(1+1.5*dt)` |

## 通用工程经验

### 1. 状态机模式
所有技能都遵循相同的状态机架构：
- `phase: 'idle' | ...` 控制技能生命周期
- `timer: number` 阶段倒计时
- `updateXxx()` 每帧推进状态
- `handleSkillActivation` 触发初始状态

### 2. 渲染层次
- 全局效果（二向箔变形、蓝色滤镜、红蓝双曝）放在 `main.ts` 的 `render()` 中
- 局部效果（陨石红圈、小猫、透镜）在各自模块渲染
- 离屏 Canvas 处理需要捕获+后处理的场景

### 3. 性能策略
- 像素级操作（透镜位移）降采样到 1/3 分辨率
- 画面叠加（量子双曝）只用半透明 tint 层，不重绘
- 变形变换（二向箔）用 Canvas 原生 `transform`，零额外开销

### 4. 粒子系统
- 所有技能复用 `state.particles` 数组
- 粒子类型通过 `color` 区分，不做新数据结构
- 每帧随机生成，用 `Math.random() < threshold` 控制密度

### 5. 演习模式一致性
- 所有 `updateXxx()` 函数 export，Practice.ts import 后直接调用
- PracticeState 镜像 SiegeState 的技能字段
- 每次新增字段需同步三处：SiegeState 接口、init、PracticeState 接口、Practice init

### 6. 库存管理
- 新指挥官加入 `MVP_COMMANDERS` 数组（Parts.ts）
- 新指挥官加入 `defaultInventory`（Inventory.ts）
- `DATA_VERSION` 每次新增零件时递增

### 困难与解决
| 困难 | 解决 |
|------|------|
| flooding 阶段水面从 MAX_H 骤降至 0.3×MAX_H | flooding 直接设 `MAX_H` 不变，无缝衔接 raining 终点 |

- 4s 悬停(白线) → 1s 加速下落(红线) → 剑中腹命中敌人中心
- 敌人越动越慢（感受到剑的重量）
- 命中瞬间：神圣金光粒子爆发（非爆炸式）
- 随机台词：3 套经典语录逐行展示

### 工程落地
- 剑绘制 (~50 行)：锥形双刃剑身+中脊线+血槽，翼型护手(红宝石)，皮革缠绕握柄，金球红宝石配重球
- 下降物理：`descend = 138 * t²`（`d = ½at²`），t = `1 - timer/0.707`，t=1 时剑腹命中 `ey`
- 锚点定位：`ey - TANK_RADIUS * 13 - 5`（12 个坦克身高），下降距离 138px
- 粒子系统：金柄粒子（`#fff8c0/#ffdd44`）+ 冷蓝剑身粒子（`#4488ff/#3355cc`）+ 金色曳尾 + 60 粒子神圣击杀

### 困难与解决
| 困难 | 解决 |
|------|------|
| 剑尖碰到敌人就消失（不是剑腹）| 下降距离从 112→138（半剑长偏移 ~25px） |
| 下落距离被改后剑柄穿过敌人 | 回到只改锚点不改距离的思路，精确计算 descend = 锚距 - 剑腹偏移 |
| 每一帧都在闪（双 reversal）| 加 `rewindReversed` 标记，只反转一次，后续帧加速同向 |
| 台词被技能结束截断 | 创建独立 `QuotePlayer` 模块，单例模式，与技能层完全解耦 |
| 台词切换时闪烁 | `alpha = 0` → `alpha = 0.05`，新行立刻有微弱可见度 |
| 蓝光粒子颜色太浅看不到 | 纯蓝系 `#4488ff/#3355cc` 替代蓝白系 |
| 爆炸像炸药不是神圣感 | 60 个细金粒子慢扩散 + smokeExpand → 金色光芒取代爆炸 |

## 10. 🎬 QuotePlayer — 独立台词系统

### 设计动机
技能的机制层（战斗效果）和台词层（文字展示）必须解耦。台词不能因为技能结束而被截断。

### 架构
```
QuotePlayer (模块级单例)
├── playQuote(lines)    ← 技能 handler 一句调用
├── updateQuote(dt)     ← main.ts update 中独立运行
└── renderQuote(ctx)    ← main.ts render 末尾独立渲染
```

### 核心设计
- 模块级 `state` 不暴露给外部，技能层不需要持有任何台词状态
- 逐行淡入（2s/行，金色 `rgba(255,220,100)`）
- 全部显示后 5s 整体淡出
- 默认模式，以后所有文学/文化类技能统一使用

## 核心经验总结

### 7. 典故优先于机制
机制是手段，故事才是目的。先找典故 → 把故事翻译成游戏事件 → 特效讲好这个故事。玩家按 E 键的"卧槽"瞬间比任何机制都珍贵。

### 8. 台词系统解耦
台词和技能必须独立运行。技能负责战斗，台词负责叙事，两者通过一句 `playQuote(lines)` 信号关联。技能结束不影响台词播放。

### 9. 视觉即故事
每个粒子都要"会说话"。木马的金光 = 希腊联军伏兵的紧张感。方舟的浪花 = 大洪水的狂暴。神圣击杀的细金粒子 = 达摩克利斯之剑的威严，不是爆炸。

### 10. 参数调优的黄金法则
- 所有距离计算都要从锚点精确推导到目标位置（不靠猜）
- 每次修改都要追溯公式，保证 `t=1` 时精确命中
- 粒子颜色不能太浅（用户看不到），不能太像烟雾（不像光芒）
