# 对战框架抽离 — 对话记录

## 用户的想法

> 围城模式可以看作是最基本的对战系统框架+基地。能否把这个最一般的对战系统直接剥离出来，围城模式只保留围城的特质即基地相关代码。以后再做任何模式，就可以直接通过新的内容与对战框架进行组装得到新的游戏模式。这是一种组合优于继承的模式。

## Claude 的回应

这个想法不仅有价值，而且是 TankEngineer 架构进化的必然方向。

### 为什么值得做

已经证明了"提取共用代码"的威力——10 个技能从 Siege.ts 拆出去，文件缩小 40%，零 bug。

当前 Practice.ts 和 Siege.ts 的 update/render 循环是两套独立代码，循环结构完全重复。每次加新功能都要改两个文件。

组合优于继承：对战框架 = 一套标准化的 update 管道 + render 管道。新模式的开发变成：

```
围城模式 = 框架 + 指挥所 + 波次 + 计时
演习模式 = 框架 + 固定靶 + 活动靶 + 重置按钮
未来PvP  = 框架 + 双玩家输入 + 计分板
未来Boss战 = 框架 + Boss AI + 阶段机制
```

### 工程可行性

Siege.ts 的 update 循环约 70% 是通用框架，30% 是围城特有。抽出一个 `BattleEngine`，围城和演习各自调用它，再追加自己的逻辑。
