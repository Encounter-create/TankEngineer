# BUG修复日志

## #1 二向箔（歌者）技能白屏卡死

**日期**：2026-07-10

**现象**：释放二向箔技能后，游戏随机出现完全卡死（画面冻结永不恢复），常发生在白屏阶段播放诗句之间。

**根因**：二向箔白屏/恢复期间敌军持续攻击指挥所(CC)。CC被摧毁→`endSiege`将phase设为defeat→`updateSiege`提前return→`updateBivector`不再运行→`bivectorPhase`永远卡在whiteout/recovering→`drawBivector`每帧继续画白屏覆盖→画面永远白屏。不是死循环，是状态机泄漏：游戏已结束但技能视觉效果未清理。

**修复**：[Siege.ts](src/modes/Siege.ts) `endSiege`函数中、[TwoKings.ts](src/modes/TwoKings.ts) phase变为defeat/victory时，强制重置bivector状态为idle。

## 经验教训

### 1. 修bug不难，难的是定位bug——把问题说清楚，问题就解决了一半

此次排查过程中，最初的描述"二向箔卡死"经过了多轮细化：从"随机卡死"→"白屏阶段"→"timer停在2.4不变"→"演习模式不卡"→"CC被摧毁导致状态泄漏"。每一步信息增益都在缩小排查范围。**把现象精确描述清楚，比写代码更重要**。

### 2. 定位程序bug不要纯做理论推测，要像做物理实验一样用console实验

本次排查的经验路径：

```
纯代码审查（无果）
→ 加帧号日志确认卡死在update层还是render层（确认：update完整执行）
→ 加CCAttack/fireZones细分日志（卡在这两者之间？不，都执行了）
→ 加render层日志（render也在正常运行！）
→ 推论：不是死循环，是"视觉冻住但逻辑在跑"→ 锁定为状态泄漏
```

每一步console.log都是实验，每个实验结果都在缩小假设空间。这符合**现象反映本质、实践促进理论**的辩证哲学——先有实验数据，再形成理论解释，而不是先猜原因再找证据。

### 3. 区分"真死机"与"视觉冻住"——不响应≠主线程阻塞

本次的核心歧义：用户说"卡死"，自然理解为死循环/主线程阻塞。但实际是游戏逻辑正常运行，视觉效果层（白屏覆盖）未清理导致的"假冻住"。**先确认JS线程是否存活**（用持续输出的console.log），是区分这两类问题的第一手段。

### 4. 对比工作/不工作的场景差异——演习模式是黄金对照组

"演习模式基本从来不卡"这条信息直接指向了问题本质。围城vs演习的差异表：

| | 围城 | 演习 |
|---|---|---|
| CC指挥所 | ✅ | ❌ |
| 敌军波次 | ✅ | ❌ |
| CC被攻击 | ✅ | ❌ |

三者叠加→CC在白屏期间被摧毁→状态泄漏。对照实验法在debug中极其高效。

### 5. 状态机过渡必须清理活跃技能——"相位变更=全清"

游戏phase（playing→defeat/victory）是顶层状态。任何技能如果跨越了phase变更而不清理视觉效果，都会导致残留。**每个顶层状态变更点都应该是所有技能的`reset()`调用点**。本次修复在`endSiege`和TwoKings的defeat/victory路径中加入了bivector重置，但理想方案应该是统一的`resetAllSkills()`函数，在各模式游戏结束时调用。

### 6. AI写的代码必须写清楚注释——方便人类和机器理解

此次排查中，`drawBivector`函数的注释只有一行`// Text drawn in main.ts`，没有说明白屏覆盖的作用和与其他系统的交互。`updateBivector`的阶段转换逻辑分布在函数各处，没有一个清晰的状态机图示。**代码注释应该覆盖：为什么这样设计、与其他系统的交互约定、状态转换图**，而不是只描述代码做了什么（代码已经自己说明了）。

### 7. 防御性编程：每个while/for循环应有明确的退出保证

排查过程中审查了所有`while`循环（Trojan.ts的while、GameLoop的while、Physics的角度归一化while等），虽然最终bug不在循环中，但这个检查习惯值得保留。**任何无界循环都是潜在炸弹**。

### 8. 诊断日志的"洋葱剥皮法"——从粗到细逐层深入

本次的日志添加顺序值得复用：
1. 先确认"死在哪一层"（update入口）
2. 再确认"死在这一层的哪一段"（updateBattle前后）
3. 继续细分"死在这一段的哪一行"（CCAttack/fireZones前后）
4. 交叉验证"另起一行看render"（证明问题不在update层）

每一层日志定位精度提升一个数量级，避免一次性加几十条日志淹没在噪音里。
