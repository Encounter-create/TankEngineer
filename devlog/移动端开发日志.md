# 移动端虚拟摇杆开发日志

## 概述

为坦克工程师添加手机端触屏操控支持，实现一份代码双端自动适配：桌面端保持键盘+鼠标，手机端显示虚拟摇杆。

**开发日期**：2026-07-10 ~ 2026-07-11

**核心设计决策**：所有改动集中在 Input 层 + 渲染层，Siege/TwoKings/Practice 三个游戏模式代码**零修改**。触摸事件在 Input.ts 中转化为虚拟按键/虚拟鼠标，游戏逻辑完全无感。

---

## 架构

```
触摸屏 → Input.ts (touch事件 → 虚拟WASD/鼠标/E键) → 游戏逻辑(不变)
                                         ↓
                              VirtualJoystick.ts (渲染摇杆UI)
```

## 修改文件清单

| 文件 | 改动内容 |
|------|---------|
| `src/core/Input.ts` | 触摸事件监听、双摇杆状态管理、坐标旋转矫正、触屏滚动、多坦切换 |
| `src/ui/VirtualJoystick.ts` | 新建，移动摇杆+攻击摇杆+EU按钮+多坦123按钮渲染 |
| `src/main.ts` | 横屏自适应(旋转90°+等比缩放)、渲染调度、battleMode切换 |
| `index.html` | canvas 居中+禁止触屏默认手势 |
| `src/systems/Practice.ts` | 补 U 键调试（BattleEngine 已处理，后撤销多余 handler） |

## 关键困难与解决方案

### 1. 分支切换导致代码丢失

在执行 `git add` 时误将 `node_modules` 暂存，试图 `git reset` 撤销时意外切换到 `gh-pages` 分支。切回 `master` 后，三个已修改但未 commit 的文件（Input.ts、main.ts、index.html）全部丢失。

**教训**：任何时候操作 git 前先 `git status` 确认当前分支和工作区状态。大规模改动应频繁 commit，不要攒到最后。

**解决**：`git checkout -- . && git clean -fd` 回到干净状态，从零重写。

---

### 2. 手机横屏旋转不响应（耗时最长的bug）

**现象**：手机竖屏打开游戏，画面极小且竖屏显示，旋转手机后 canvas 不跟随变化。

**尝试过的方案**（均失败）：
- CSS `@media (orientation: landscape)` + `max-width/max-height` → 不生效
- JS `window.addEventListener('resize', ...)` + `orientationchange` → 事件不触发或延迟
- `requestAnimationFrame` 轮询视口尺寸 → 可行但过度设计
- CSS `transform: rotate(90deg)` 位置计算反复出错

**最终方案**：JS `fitCanvas()` 在竖屏时将 canvas 旋转 90° + 等比缩放。CSS 去掉所有尺寸约束，全交 JS 控制。`document.documentElement.clientWidth/Height` 替换 `window.innerWidth/Height`（后者在 iOS Safari 不可靠）。

```javascript
// 竖屏：canvas 高度 = 手机宽度，宽度按比例，旋转90°
const w = vw * ratio;
canvas.style.width = w + 'px';
canvas.style.height = vw + 'px';
canvas.style.transform = 'translate(-50%, -50%) rotate(90deg)';
```

**教训**：移动端视口尺寸获取不能依赖单一 API。CSS 和 JS 同时设置 transform 会互相覆盖，必须只在一处处理。

---

### 3. 旋转后触摸坐标映射错误

**现象**：canvas CSS 旋转 90° 后，按钮视觉位置和触摸生效位置完全偏移。

**根因**：`getBoundingClientRect()` 返回旋转后的包围盒，但视觉宽（原 CSS 高）应对应 buffer 高(704)、视觉高（原 CSS 宽）应对应 buffer 宽(960)。初版把映射因子搞反了。

**最终公式**：
```typescript
if (r.width < r.height) { // canvas 被旋转了90°
  const vx = (cx - r.left) * canvas.height / r.width;  // 视觉宽 → buffer 高
  const vy = (cy - r.top) * canvas.width / r.height;   // 视觉高 → buffer 宽
  return new Vec2(vy, canvas.height - vx);              // 交换XY + 反转Y
}
```

**教训**：涉及坐标变换的 bug，手算验证中心点映射是否正确（canvas中心 (480,352) 应对应视觉中心），一算就对，不对就说明公式反了。

---

### 4. 非战斗界面按钮全部失效

**现象**：主菜单、大厅、车间所有按钮点击无反应。

**根因**：触摸分区（摇杆区、开火区）覆盖了整个底部，非战斗界面的按钮被错误分类为摇杆/开火区，不会触发鼠标点击。

**解决**：新增 `battleMode` 标志。非战斗界面 (`setBattleMode(false)`) 时所有触摸统一归为 'aim' 区，模拟标准鼠标点击。战斗界面才启用摇杆/开火/技能分区。

---

### 5. 移动摇杆方向错误（只能向下偏转）

**现象**：摇杆拖拽时坦克只往下方移动。

**根因**：`joystickDir` 存储的是触摸在 canvas 上的绝对坐标(如 110,500)，而非相对于摇杆中心(110,554)的方向向量。`getMoveDir()` 直接返回绝对坐标，经 `norm()` 归一化后方向完全错误。

**解决**：`joystickDir = touchPos.sub(joystickCenter).norm()`，存储归一化方向向量。同时分离 `joystickRaw`（原始偏移，供渲染器画拇指位置）和 `joystickDir`（归一化方向，供游戏移动）。

---

### 6. 攻击键改为拖拽摇杆

**初版**：右下角固定红色按钮，按住开火，但不能控制方向。

**改进**：攻击键改为与移动摇杆相同的可拖拽设计。拖拽方向控制炮塔瞄准，按住持续开火。两个摇杆各司其职：左=移动，右=瞄准+开火。

实现：右侧摇杆拖拽时设置 `mousePos = fireCenter + fireRaw * 10`，将方向映射到远处点，炮塔跟随该点旋转。

---

### 7. U键双击消

**现象**：U 键按一次能开，再按关不掉。

**根因**：三个地方处理 U 键——BattleEngine（Siege/ 演习）、TwoKings 自己的 update、以及 main.ts 新增的移动端 handler。触摸一次触发多个 handler，互相抵消。

**解决**：删除 main.ts 中的移动端 U 键处理。触摸分区中的 'u' 区直接向 `justPressed` 注入 `'KeyU'`，由各模式已有的 U 键处理器统一响应。Practice.ts 不需要额外加（它走 BattleEngine）。

**教训**：在已有键盘处理的基础上加触摸支持，应尽量复用现有按键检测路径，而不是新增独立 handler。

---

### 8. 演习模式 U 键无效

**现象**：Practice.ts 未处理 U 键，但用户说"原本有效"。

**根因**：Practice 模式使用 `updateBattle`（BattleEngine），BattleEngine 第 35 行已有 U 键处理。我误以为 Practice 没有，擅自加了一个，又造成双击消。

**解决**：删掉多余的 handler。

---

### 9. 手指滑动翻页（文本查看器）

在主菜单的新手教程/制作团队面板中，需要手指滑动即可翻页。

**解决**：在 Input.ts 的 aim 区 touchmove 中累加 Y 轴偏移，通过 `consumeTouchScroll()` 供 main.ts 消费，与鼠标滚轮合并后驱动文本滚动。

## 经验教训

1. **一份代码双端适配的核心是 Input 抽象层**。所有平台差异封装在 Input 类中，游戏逻辑不感知设备类型。

2. **移动端坐标变换是最大的坑**。CSS 旋转 + canvas 缩放 + getBoundingClientRect 三者的交互极其容易出错。手算验证比反复调参高效得多。

3. **git 操作前务必确认分支和状态**。大规模未提交改动时禁用 `git reset --hard` 等危险操作。

4. **触屏 UI 需要比鼠标更大的点击区域**。桌面 16px 半径够用，手机需要 35-45px。所有按钮和分区阈值在开发时就要考虑拇指尺寸。

5. **复用已有键盘路径，不要新建独立 handler**。`justPressed.add('KeyX')` 比新增 `if (touchU) toggle()` 更可靠、更一致。

6. **先锁定横屏，再做适配**。横竖屏自适应是伪需求——没有玩家会在竖屏玩横版游戏。强制横屏（旋转锁定 + canvas 旋转 90°）是正确的做法。

7. **测试要覆盖所有模式**。围城、双王、演习三个模式各有不同的按键处理路径，每个都要测到。
